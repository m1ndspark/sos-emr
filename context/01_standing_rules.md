# Standing Rules (EMR)

Source: SOS EMR Standing Rules (Updated Master), plus project boilerplate.
These govern how code work is done. Rules override any contradicting guidance
from older session notes.

PLATFORM AND LANGUAGE
- Zoho Creator, Deluge only. Confirm the live Creator build before app work
  rather than assuming a fixed version. Creator behavior wins over generic docs.

CODE DELIVERY
- No comments in code unless asked.
- Always show the full function, never partial snippets or diffs unless asked.
- Forms and workflows are a single unit. Evaluate any field change against all
  known workflows and propose the matching workflow change at the same time.

HANDLING UNCERTAINTY
- Never guess. Stop and ask if anything is unclear.
- Ask before assuming form names, field names, data types, relationships, intent.
- Flag risks clearly, then continue unless the risk is blocking.
- Never describe Creator UI from memory. Ask for screenshots or doc links.

KNOWN SYNTAX RULES (DO NOT REPEAT THESE MISTAKES)
- show() and hide() are valid only in On User Input actions.
- Field-level display rules use condition expressions only, not procedural Deluge.
- The void keyword is only for standalone functions in the Functions tab.
- Never generate IDs inline in On User Input. ID generation is a separate workflow.
- Stamp generators are always separate workflows, never bundled into show/hide.
- Confirm the exact field link name before writing Deluge that references it.

DATA INTEGRITY
- All object IDs are read-only to all users, generated by workflow, never editable.
- All medical coding is ICD-10. ICD-9 is retired and never valid.
- Record locking is required on Sequence_Tracker reads to prevent duplicate IDs.
  Critical before go-live. (See open contradiction on locking scope.)
- Field deletion order: remove all workflow references first, then delete the
  field. Use Creator's View Field References on every field before deletion.

USER AND IDENTITY MODEL
- Identity resolved by email domain: @sosmmc.com is Admin, @sosreferrals.com is
  Employee portal, all other domains are Partner portal.
- Resolution centralized in the fn_resolveUserIdentity standalone function. Call
  it from form workflows rather than duplicating domain logic.
- Portal users create entries on forms, not forms themselves.

PORTAL ARCHITECTURE
- Custom pages, not default Creator nav, to avoid per-user licensing costs.
- Partner portal user creation is a manual two-step process (create portal user
  in Creator settings, then create the matching PartnerPortalUsers record, same
  email). This is a permanent Creator limitation and cannot be automated.

MOBILE-FIRST
- All employee-facing interfaces are mobile-first. Minimum tap target 44x44 px.
  Test on a phone viewport before go-live.

SAFETY AND PRIVACY
- No PHI in external AI dictation fields. Patient identity is stamped by the form
  on load and never passed to external AI services.
- No PHI in SMS bodies. Keep PHI out of QuickBooks entirely. PHI lives in the EMR
  and Zoho Books, both BAA-covered.
- HIPAA compliance confirmation required before go-live of any AI feature.
